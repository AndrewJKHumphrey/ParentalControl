Developer Guide

Internal reference for building, testing, and extending ParentGuard. Not intended for end users.

Project Structure

The solution contains six C# projects plus the browser extension and WiX installer.

ProjectTypeDescription
ParentalControl.Corenet10.0Shared models, EF Core DbContext, IPC message types. Referenced by all other C# projects.
ParentalControl.Servicenet10.0-windowsWindows background service. Enforces all rules (screen time, app blocking, focus mode). Hosts the named-pipe IPC server.
ParentalControl.Watchdognet10.0-windowsLightweight watchdog service. Restarts the main service if it crashes; the main service reciprocates.
ParentalControl.UInet10.0-windows WPFAdmin UI. Password-protected. Communicates with the service over the named pipe.
ParentalControl.NativeHostnet10.0-windowsNative messaging host launched by the browser extension. Bridges the extension to the service over the named pipe.
ParentalControl.ExtensionBrowser extension (JS)Chromium extension. Intercepts navigations, queries the NativeHost for the current rule set, shows the block page.
ParentalControl.InstallerWiX v4Produces ParentGuard-Setup.msi. Harvest WXS files are auto-generated by build-installer.ps1.

Dev Workflow

Debug builds (day-to-day development)

Build and run directly from Visual Studio or the CLI. The Debug build skips the login window so you can iterate quickly.

# Build all projects
dotnet build --configuration Debug

# Run the UI (no elevation needed)
dotnet run --project ParentalControl.UI
Note: Debug builds expose extra UI — the Raven and Custom themes, a Debug section in Settings, and the Developer Guide nav button — all hidden in Release.

Testing the full Release stack locally

Use run-ui-release.bat to publish the UI in Release mode and launch it without a full service reinstall. The service must already be running (installed via install.ps1 or the MSI).

# From the solution root — no elevation required
run-ui-release.bat

This publishes to publish\ui-release-test\ and immediately launches the exe, giving you the full login flow and Release-only behaviour.

Installing for testing (dev machine)

Use install.bat (which wraps install.ps1) when you need the service and watchdog running on your dev machine. It publishes all four projects, stops any running services, copies files to Program Files, and re-registers the services.

# Must be run as Administrator — will self-elevate if not
install.bat
Warning: This overwrites the install directory. Stop the UI and close browsers before running to avoid file-lock errors.

Building the MSI for distribution

# Must be run as Administrator
.\build-installer.ps1

# Output:
#   ParentalControl.Installer\bin\Release\en-US\ParentGuard-Setup.msi

The script: publishes all four projects → merges into publish\combined\ → packs the browser extension as a .crx → auto-generates WiX harvest WXS files → builds the WiX project.

Prerequisite: extension.pem must exist in the solution root for .crx packing to succeed. If it is missing the script will skip packing and warn.

Service management (quick commands)

# Check service status
Get-Service ParentalControlService, ParentalControlWatchdog

# Start / stop manually
Start-Service ParentalControlService
Stop-Service  ParentalControlService -Force

# View recent service log entries
Get-WinEvent -FilterHashtable @{LogName='Application'; ProviderName='ParentalControl'} -MaxEvents 20 | Format-List

Known Quirks

Build error — DLL locked / error code 0x800711C7

Symptom: dotnet publish or a file copy fails mid-run with an access-denied or sharing-violation error referencing ParentalControl.Core.dll or another output DLL. Error code 0x800711C7 is Windows Smart App Control or Defender quarantining a newly compiled binary before it can be written.

Cause: Windows Smart App Control (or real-time Defender scanning) intercepts and locks freshly compiled DLLs the moment they land on disk, before the publish step finishes copying them.

Fix already applied in install.ps1: Defender exclusions for all build output directories (publish\, each project's bin\, and the install directory) are added before dotnet publish runs. If you hit this in a plain dotnet build, add the exclusion manually:

Add-MpPreference -ExclusionPath "C:\Users\<you>\ParentalControl" -ErrorAction SilentlyContinue
Note: Smart App Control cannot be disabled per-directory — if exclusions stop working, disable SAC in Windows Security → App & Browser Control → Smart App Control.

Service or UI exe locked during reinstall

Symptom: install.ps1 fails to copy ParentalControl.Service.exe or ParentalControl.UI.exe because a process holds the file open.

Fix: install.ps1 stops services, waits for processes to exit, and kills any lingering instances before copying. If you see this error outside of that script, manually stop the service and close the UI before retrying:

Stop-Service ParentalControlWatchdog, ParentalControlService -Force
Stop-Process -Name ParentalControl.UI -Force -ErrorAction SilentlyContinue

Named pipe — UI shows "Stopped" immediately after service starts

Symptom: The sidebar shows the service as stopped even though Get-Service reports it running. Refreshes after a few seconds fix it.

The IPC client has a short connection timeout on first connect. The service takes a moment to initialise its pipe server after startup. The UI retries automatically — this is cosmetic only and resolves within a few seconds.

Debug build skips the login window

In a #if DEBUG block in App.xaml.cs, the login window is bypassed so you can iterate on UI pages without re-entering the password. The full authentication flow is only active in Release builds. Use run-ui-release.bat to test authentication without doing a full service reinstall.

NativeHost manifest must be re-registered after a reinstall

The NativeHost communicates with the browser extension via the native messaging protocol. Its path is stored in a registry key under HKCU\Software\Google\Chrome\NativeMessagingHosts (and the Edge equivalent). install.ps1 re-writes this key on every run. If the extension stops communicating after a reinstall, re-run install.ps1 or check the registry path.

EF Core — "table already exists" on startup

The service uses EnsureCreated(), not EF migrations. New columns are added via raw ALTER TABLE … ADD COLUMN IF NOT EXISTS statements in Worker.EnsureDatabase(). If you add a new property to a model, add the matching ALTER TABLE there — do not delete the database and rely on EnsureCreated() alone, or existing user data will be wiped.

Database

SQLite database at C:\ProgramData\ParentalControl\data.db, managed via EF Core.

Schema changes

There are no EF migrations. To add a column:

  1. Add the property to the model class in ParentalControl.Core.
  2. Add a corresponding ALTER TABLE … ADD COLUMN … DEFAULT … in Worker.EnsureDatabase() in ParentalControl.Service.

The IF NOT EXISTS form is safe to run on every startup — it is a no-op when the column already exists.

Resetting for a clean test

# Stop services first so the file is not locked
Stop-Service ParentalControlWatchdog, ParentalControlService -Force
Remove-Item "C:\ProgramData\ParentalControl\data.db" -Force
Start-Service ParentalControlService   # watchdog starts automatically
Warning: This wipes all profiles, rules, app time data, and activity history. The default password parent1234 is restored on first run.

Inspecting the database directly

Use DB Browser for SQLite or any SQLite-compatible tool to open data.db directly. Stop the service first to avoid a sharing conflict.

Adding Themes

Themes are defined in App.xaml.cs as a dictionary of 8-colour tuples and exposed via ComboBoxItem entries in SettingsPage.xaml.

Colour tuple format

// Tuple order: AppBg, SidebarBg, CardBg, CardHover, AltRowBg, BorderColor, TextPrimary, TextSecondary
["My Theme"] = (("#0A0C20", "#060814", "#10163A", "#182248", "#0E1228", "#FFDD22", "#44DDFF", "#1AAACC"),   // dark
               ("#EEF0FF", "#E0E4FF", "#F5F8FF", "#C8D0FF", "#E8EEFF", "#997700", "#003388", "#002266")), // light

Adding a DEBUG-only theme

Mark the ComboBoxItem in SettingsPage.xaml with Tag="MonHun" Visibility="Collapsed" (for the Mon Hun group) or give it an x:Name and show it in the #if DEBUG block in SettingsPage.xaml.cs. The Mon Hun Inspired section is currently entirely DEBUG-only and revealed via a foreach loop over items tagged "MonHun".

DEBUG-only themes currently hidden in Release

  • Raven — shown via RavenThemeItem.Visibility in the DEBUG block
  • Custom — shown via CustomThemeItem.Visibility; reveals the hex colour picker panel
  • Mon Hun Inspired (25 themes) — all Tag="MonHun", pending palette review before promotion to Release

Packing the extension manually

# Edge must be installed; extension.pem must exist in the solution root
& "C:\Program Files (x86)\Microsoft\Edge\Application\msedge.exe" `
    --pack-extension=".\ParentalControl.Extension" `
    --pack-extension-key=".\extension.pem"

The output ParentalControl.Extension.crx is placed next to the solution root. build-installer.ps1 moves it into the extension directory automatically.

Keep extension.pem private. It is the signing key for the extension. Its extension ID is lackpoggaaeodfcagkfcglokeilcfokg — this ID is derived from the key and must remain stable across updates.

Native messaging host

The NativeHost is a .NET console app launched on demand by the browser when the extension requests it. Communication is via stdin/stdout with length-prefixed JSON messages (the native messaging protocol). The NativeHost connects to the same named pipe as the UI to retrieve rules and theme data.

Its registry manifest path is written by install.ps1 and configure.ps1 to both the Chrome and Edge native messaging host registry keys.

IPC & Service Architecture

Named pipe

The service hosts a named pipe server. Both the UI and the NativeHost connect as clients. Messages are length-prefixed JSON. The IpcCommand enum (in Core) defines all message types — ReloadRules is the most common, sent by the UI after any rule save to tell the service to re-read the database without restarting.

Rule enforcement loop

The service worker runs on a 5-second tick. Each tick it: checks screen time, checks focus mode windows, calls ProcessMonitor.EnforceRules() which iterates running processes and kills any that violate the active profile's rules.

Profile resolution

The service detects the active Windows session user via WTSQuerySessionInformation and matches it case-insensitively against UserProfile.WindowsUsername. If no match is found, the default profile (Id = 1, WindowsUsername = "") applies.

Adding a new IPC command

  1. Add a value to IpcCommand in ParentalControl.Core.
  2. Handle it in the switch in IpcServer.HandleCommand() in ParentalControl.Service.
  3. Call _ipc.SendAsync(IpcCommand.YourCommand) from the UI page after the relevant DB write.
